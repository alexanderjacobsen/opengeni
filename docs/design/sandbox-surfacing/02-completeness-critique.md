# Completeness Critique ‚Äî OpenGeni Sandbox Surfacing

What is still MISSING or UNDER-SPECIFIED to ship the full vision. Each gap names where it belongs.

## A. Components no module owns (orphaned responsibilities)

**A1. The negotiation‚Üíspawn workflow path is described two contradictory ways and fully owned by neither.** The integration spec (Part D, flow b) routes a viewer attach through a *short-lived* `sandboxOwnerRpcWorkflow` (executeWorkflow-awaited). But channel-B ¬ß6 routes it through `signalWithStart` on the *long-lived* `sessionWorkflow`, and the lease/owner modules route turns through the same long-lived workflow's claim loop. **No module owns the definitive answer to: does a viewer attach start/signal the session workflow, or spin a separate RPC workflow?** These have different liveness, different `continueAsNew` interactions (multi-day sessions, `session.ts:111`), and different deadlock profiles (the long-lived loop blocks for days inside `runTurn`). **Belongs in: owner module ¬ß6, as the single canonical "viewer/RPC dispatch" design**, with channel-A's `sandboxOwnerRpcWorkflow`, channel-B's `openStreamRequest`, and channel-A's FS/Git RPC all unified onto ONE mechanism. Right now three modules invent three.

**A2. The `sandboxOwnerRpcWorkflow` lifecycle, id scheme, and owner-liveness guarantee are unspecified.** The integration spec leans on it as the universal request/response carrier (CR12) but no module defines: its workflow id (`sandbox-owner-rpc::<sid>::<rpcId>`? reuse policy?), how it guarantees the owner is warm before running the activity (it acquires a viewer holder? who releases it?), its timeout, or what happens when the owner is mid-re-election. **Belongs in: a new short section in the owner module**, since it's the load-bearing seam for channel-A (FS/Git/PTY), channel-B (negotiation), and recording control.

**A3. The reaper's SECURITY DEFINER sweep function is named but never specified.** CR5 mandates an `opengeni_private` SECURITY DEFINER function to select stale `(workspace_id, session_id)` tuples cross-workspace, then per-workspace `withWorkspaceRls` loops. **No module gives its DDL, its exact selection predicate (stale viewer holders + warming-death + draining-grace in one pass), or the migration it lands in.** This is the only thing that makes liveness GC work, and it's a hole in every module (lease assumed per-workspace, owner assumed a non-existent BYPASSRLS role). **Belongs in: lease module, migration `0017`**, with the function body + the Temporal Schedule driver that calls it (the driver is also unowned ‚Äî see A4).

**A4. The reaper's Temporal Schedule driver has no owner.** CR5 says "a Temporal Schedule (`reapSandboxLeases`, every `sandboxReaperPeriodMs`)" drives it, but no module specifies the Schedule creation (where in `apps/worker/src/index.ts`?), the activity that wraps the sweep, idempotency across overlapping ticks, or what happens if a tick runs long. **Belongs in: owner module (worker wiring) + lease module (the activity body).**

**A5. The `ownerHeartbeat` activity is referenced by 4 modules but specified by none completely.** Lease, owner, crosscut (warm-meter), and channel-B (viewer liveness) all depend on it. Owner ¬ß6.4 sketches it but the adversarial review (owner B5) showed the sketch is broken (heartbeats every 60s against a 30s timeout; `sleepCancellable` undefined; self-evicts on first tick because the owner registered no holder). **No corrected canonical version exists.** It must: heartbeat ‚â§10s regardless of keep-alive cadence, refresh `expires_at` *without* requiring a holder row (epoch-keyed `touchLeaseExpiry`, owner C4), accrue `sandbox.warm_seconds` atomically (crosscut M3), and self-evict on epoch mismatch. **Belongs in: owner module, rewritten, with crosscut's metering folded in.**

## B. Interfaces referenced but never defined

**B1. The owner‚Üîruntime "owned session" decoration contract is unresolved.** Owner ¬ß7.2 picks option (B) (re-apply agent-dependent decorators per-run), but the adversarial review (owner D1‚ÄìD4) found: passing both `client`+`session` triggers SDK end-of-run serialization; `applyManifestToProvidedSession` *throws* on any root/env mismatch across agents sharing the box; the repo-clone hook re-clones every turn; and 6080 isn't wired into the constructor. CR19 acknowledges these but the *resolved* contract ‚Äî exactly what `runAgentStream`'s owned branch passes and which decorators run ‚Äî is still a list of "must fix," not a spec. **Belongs in: owner module ¬ß7, as concrete code**, because the entire ownership inversion (the architecture's foundation) depends on it and it currently doesn't compile/run.

**B2. `establishSandboxSessionFromEnvelope` ‚Äî the single recovery primitive ‚Äî depends on a non-importable SDK helper.** Owner A7/D2 flagged `isProviderSandboxNotFoundError`/`assertResumeRecreateAllowed` live in `@openai/agents-extensions/sandbox/shared`, which is **not an exported subpath**. The "never `create()` on resume-conflict, only on NotFound" rule ‚Äî the thing that prevents a second box ‚Äî has no implementable NotFound discriminator. **No module re-specifies how OpenGeni detects provider-NotFound per backend.** This is load-bearing for flows (a), (d) and the split-brain guarantee. **Belongs in: owner module + providers module (per-provider error-shape inspection).**

**B3. The capability-descriptor `workspaceRoot` / `nativeBucketMount` consumer contract is half-defined.** Providers M3/¬ß7.2 add these fields to the descriptor so `environment.ts` (HOME) and `agent-turn.ts` (signed-download) branch on data not backend names ‚Äî but the actual mount-strategy decision (which storage√óbackend needs signed download vs native mount) is "owned by the file-resource module" which **does not exist in the 10 modules.** The os module also adds `workspaceRoot` and they conflict on ownership. **Belongs in: providers module, as the authoritative descriptor field + a stated rule** that the env/mount code consumes it; the os module's override must defer to it.

**B4. The `SessionCapabilities` ‚Üí `ClientConfig` relationship for desktop advertisement is doubly-defined and never reconciled.** Client ¬ß1.6 adds a `sandbox` block to `ClientConfig` (deployment-wide: `allowedBackends`, `desktopCapableBackends`); channel-A adds `structuredServices`; channel-B adds `desktopStream.supported`. The integration spec's A.4 unifies the *per-session* doc but never says what the *deployment-wide* `ClientConfig` carries or how `desktopCapableBackends` is computed. Client M2 flagged the `structuredServices` extension is never consumed by any route. **Belongs in: client module, as the single `ClientConfig.sandbox` shape** + the config handler that fills it (intersect configured backends with the contracts descriptor table).

**B5. Per-provider `build()` field mappings are specified for Modal only.** Providers G1 found e2b (`timeout` in ms not `timeoutMs`, no `apiKey` field ‚Äî env-only), daytona (`timeoutSec`/`exposedPortUrlTtlS`), runloop (nested `timeouts.keepAliveTimeoutMs`), vercel (`timeoutMs` from seconds) all diverge in field names, units (sec vs ms), and shape (top-level vs nested). "The others mirror Modal exactly" is false. **No module gives the 6 other `build()` bodies.** Without them the registry doesn't compile. **Belongs in: providers module ¬ß3, one block per provider, each pinned to its `.d.ts`.**

**B6. The desktop stream URL assembly contract contradicts itself across channel-B, channel-A, desktop-image, and client.** `urlForExposedPort` returns no path (verified, channel-B C3/desktop-image C3); the `/vnc.html` + `?path=websockify` must be appended client-side; Blaxel's token is `bl_preview_token` not `token` (client C3). The integration spec (CR15) says "appended client-side" but **no module specifies the single canonical `buildStreamUrl` (server) + `desktopSocketUrl` (client) pair** ‚Äî channel-B's server version and client's `desktopSocketUrl` still disagree on whether the OpenGeni token is appended. **Belongs in: channel-B module (server) + client module (`desktop.ts`), as one reconciled pair.**

## C. Failure modes unhandled

**C1. Concurrent-write conflicts on the file system are explicitly punted but the FS write path has no conflict story at all.** Channel-A's `fsWrite` via `createEditor` requires synthesizing an apply-patch diff from a read-modify-write (channel-A C4) ‚Äî that's a **lost-update race** between two viewers, or a viewer and the agent, with no detection. The matrix says "last-writer-wins, both get `fs.changed`" but the apply-patch *diff will fail to apply* if the file changed underneath (the diff's context lines won't match), surfacing as an opaque error not last-writer-wins. **Belongs in: channel-A module**, which must either gate writes on the `revision` token (optimistic concurrency) or specify the diff-apply-failure ‚Üí re-read ‚Üí retry loop.

**C2. The `revision` counter's reset-on-re-election breaks client cache invalidation (unresolved).** Channel-A H3/crosscut found the counter resets to 0 on owner re-election; client's "skip re-fetch until revision exceeds cached" then *ignores all changes* until it climbs back. The integration spec doesn't fold this in. **Belongs in: channel-A module**, pairing `revision` with `lease_epoch` into a `(epoch, revision)` tuple the client compares.

**C3. PTY reattach / scrollback replay for a late viewer is missing.** Channel-A M6: the whole point of persisting `sandbox_pty_sessions` is reattach, but `useTerminal` always opens fresh and there's no read path for an existing `ptyId`'s backlog. A second viewer of a shared PTY sees nothing. **Belongs in: channel-A module** (a `pty/attach` that replays buffered `terminal.pty.output.delta` from `session_events` via SSE gap-fill).

**C4. Recording finalize across owner re-election / box rollover loses the artifact.** Recording's own matrix admits "recordings do not survive owner re-election in v1" and the eager `rm` before confirmed upload (recording F9) makes failed uploads unrecoverable. Flow (d) says on-turn recordings "auto-restart" but the *in-flight* file on the old box is gone. **Belongs in: recording module**, which needs a flush-to-storage-before-rollover hook tied to the owner's rollover timer (the owner already persists the envelope there ‚Äî recording must piggyback).

**C5. Provider fan-out cap (R6) has no enforcement path, only a test.** Crosscut/channel-B specify the load test (the GA gate) and a `maxViewersPerSession` config, but **no module specifies where the cap is enforced** ‚Äî it must be in `acquireLease(kind:"viewer")` (reject when `viewer_holders >= cap`), returning a typed `429` the negotiation route surfaces. The lease module's acquire critical section doesn't mention it. **Belongs in: lease module (acquire) + client module (429 handling).**

**C6. Grant-revocation ‚Üí live-socket teardown is specified for the token-file but not the live VNC connection.** Crosscut 1.5 reaps the holder; channel-B 7.4 says revoke also "bounces the live VNC client (x11vnc -disconnect/pkill)." But **no module owns the mechanism that maps a revoked `viewerId` to the live x11vnc client connection to disconnect it** ‚Äî deleting the websockify token file only blocks *reconnect* (channel-B's own finding that websockify caches per-version). A revoked viewer keeps watching until their socket drops. **Belongs in: channel-B module + owner module** (owner needs a `revokeViewer(viewerId)` that disconnects the live RFB session, not just the token).

**C7. The "spawner died mid-resume but a viewer is waiting" path has an up-to-120s stall with no client feedback.** Owner E1: a viewer that `acquireLease` sees `warming` polls until `warmingTtl` (120s) before the reaper resets it. Client's `useStreamCapabilities` polls every 1.5s but the matrix only says "surface an error after ~30s." The two disagree on the deadline and **no module specifies the user-facing "still warming, this can take up to N seconds" contract.** **Belongs in: client module + lease module** (align the poll deadline to `warmingTtl`).

## D. Provider quirks not threaded

**D1. runloop/vercel PTY-incapability degrades Terminal but the agent-driven computer-use also needs PTY-less handling.** The descriptor marks `Terminal.pty:false` for runloop/vercel, but computer-use's `Computer` impl and channel-A's PTY both assume exec works uniformly ‚Äî and channel-A C1 found `session.exec` doesn't exist on *any* provider (only `execCommand` returning banner-wrapped strings). CR13 resolves this for channel-A but **computer-use's CR14 resolution and channel-A's CR13 resolution must use the same banner-stripping parser** (`sandboxCommandExitCode`), and neither module cross-references the other. **Belongs in: a shared `execAndParse` helper in runtime**, consumed by both computer-use and channel-A, specified once.

**D2. Blaxel on-demand ports + scoped `bl_preview_token` interact with token rotation in an unspecified way.** Blaxel is the only `supportsOnDemandPorts:true` backend; its preview token has its own TTL embedded in the URL query. When the owner rotates the data-plane URL (every ~60s), it must re-call `previews.createIfNotExists` and the *provider* token changes too ‚Äî but channel-B's rotation only re-mints the *OpenGeni* token. **Belongs in: channel-B module**, threading per-provider URL re-resolution into the rotation timer (Daytona signed-preview TTL has the same issue).

**D3. Modal `exposedPorts` reads `dockerExposedPorts` (cross-named) ‚Äî the fix is stated but the e2b/daytona/runloop/blaxel branches don't say where 6080 comes from.** CR10 fixes Modal but providers ¬ß3.4 / channel-B / desktop-image each gesture at 6080-injection differently. **Belongs in: providers module**, as the single `createSandboxClient` port-merge rule applied uniformly across all desktop-capable branches (with runloop now `supportsOnDemandPorts:false` per CR9, so it needs injection too).

**D4. Vercel's 5h hard cap + Modal's 24h cap need a *generic* lifetime-ceiling rollover, but only Modal's is specified.** Flow (d) covers Modal's 24h snapshot-rollover; vercel's 5h cap (`hardLifetimeMs`) hits the same wall sooner and vercel is headless (no desktop) but a terminal viewer could hold it. The owner's rollover timer reads `descriptor.lifetime.hardLifetimeMs` but **no module specifies the headless-rollover path** (no displayStack to re-establish, but the envelope/epoch/URL-rotation still apply). **Belongs in: owner module**, generalizing the rollover beyond Modal-desktop.

## E. Client/SDK gaps

**E1. The SDK `ClientConfig` mirror does not exist (client C1 ‚Äî won't compile).** `getClientConfig(): Promise<ClientConfig>` references an SDK type with zero matches in `sdk/types.ts`. The full mirror (all existing fields + the new `sandbox` block) must be specified and parity-pinned. **Belongs in: client module ¬ß2.**

**E2. The proxy pattern for the new methods is asserted "like streamEvents" but streamEvents is a bespoke SSE re-stream, not JSON passthrough (client #10).** A 3rd-party proxying `negotiateStream`/`heartbeatViewer` needs plain JSON passthrough handlers, and must pass the desktop `url`/`token` *verbatim* (different origin than their API). **Belongs in: client module ¬ß5**, as explicit proxy guidance distinct from the SSE helper.

**E3. SSR/no-DOM handling for noVNC/xterm/Pierre is mentioned per-component but there's no unified lazy-import + hydration-placeholder contract.** Four components dynamically import browser-only deps; the failure matrix has one SSR row but no module specifies the shared pattern (gate behind `useEffect`, render placeholder until hydration, optional-peer-absent fallback). **Belongs in: client module ¬ß3**, as one stated convention.

**E4. The viewer heartbeat `workspaceId` provenance and the `OpenGeniApiError`-on-409 handling are unspecified (client #5, #9).** Hooks take only `sessionId` but methods need `workspaceId` (from context); `heartbeatViewer` throws `OpenGeniApiError` on 409, so the hook must catch-and-branch on `.status`, not read a result field. **Belongs in: client module ¬ß3.**

## F. Data-shape mismatches

**F1. `LeaseRow`/`mapLeaseRow` is referenced ~15√ó but the integration spec only says "defined once in db" ‚Äî the dual-shape problem (raw snake_case vs Drizzle camelCase from `readLease`) is noted (lease M2) but the resolution isn't in any module as code.** **Belongs in: lease module.**

**F2. The recording byte-transfer shape violates Temporal payload limits (recording F10/crosscut).** `readRecordingBytes` returns a 256MB `Uint8Array` through an activity result. CR14 says "inside the owner activity, never serialized" but **no module specifies that the finalize activity does read+PUT in one invocation on the owning worker** vs. the cross-queue routing every other owner call uses. This is a genuine architectural tension (owner methods route via per-session queue; this one must NOT serialize its result). **Belongs in: recording module + owner module.**

**F3. `durationSeconds`/`sizeBytes` on `RecordingAvailablePayload` are never computed (recording F14).** No `ffprobe` call, no wall-time tracking. **Belongs in: recording module.**

**F4. The SDK mirror of recording/channel-A payloads is "mirror the Zod" but the SDK is plain-TS-no-Zod (recording F15).** Every module that says "mirror in sdk/types.ts" must specify the plain-TS shape, not "mirror the Zod object." **Belongs in: each contract-touching module's SDK section** ‚Äî a systemic instruction the integration spec should state once.

## G. Migration / ops gaps

**G1. The drizzle journal + snapshot requirement is stated for 0017 but the full migration ordering (0017‚Äì0021) is asserted without the journal-append mechanics for the hand-written RLS/SECURITY-DEFINER SQL.** Drizzle-kit `generate` won't produce the RLS GRANT blocks or the SECURITY DEFINER function; those are hand-authored and must append `_journal.json` + a snapshot manually or they never run (crosscut M1, multiple modules). **Belongs in: a stated migration-authoring convention** (probably lease module, as the first/most-complex migration), since every module gets this wrong.

**G2. `delegationSecret` is optional but required-when-desktop-enabled ‚Äî the boot validation is named (crosscut C4) but not the feature-disable fallback.** What happens when desktop is configured but the secret is absent ‚Äî hard boot-fail or graceful `DesktopStream.transport:null`? **Belongs in: config module + crosscut.**

**G3. The `sandboxOwnershipEnabled` rollback path is asserted reversible but the *data* left behind isn't addressed.** Flipping back to per-run leaves orphaned `sandbox_leases` rows in `warm`/`draining` and live boxes with no owner. **No module specifies the drain-on-disable / orphan-lease cleanup.** **Belongs in: crosscut rollout + owner module.**

**G4. Deployment per-provider required-env blocks read non-existent `SandboxSpec` fields (providers G2).** The `SANDBOX_REQUIRED_ENV` table getters reference `s.modalAppName` etc. that don't exist on `SandboxSpec` (only `backend`/`preparationProfiles`/`envAllowlist`). The values must come from the `env` record, and the table must fit *both* call sites (`requiredRuntimeEnvVars` string-list and `buildRuntimeEnv` env-getter). **Belongs in: providers module ¬ß6 / deployment.**

**G5. No observability/metrics for the new surfaces.** Route-label patterns aren't updated (client #7), so the new routes bucket as `unknown`. No metric for `owner.spawn` count vs distinct `(session,epoch)` (the double-spawn detector the Phase-1 exit gate *requires*). No warm-box gauge, no fan-out-per-port gauge. **Belongs in: crosscut (the exit-gate metrics) + each route module (label patterns).**

## H. Untested risks (gaps in the test plan itself)

**H1. The split-brain / epoch-fence has unit tests (lease T5) but no integration test exercises the *heartbeat-path* fence (CR3) ‚Äî the exact bug the review found.** The test plan tests epoch on turn-arrival, not on `heartbeatLeaseHolder`, which is where a superseded owner actually keeps the box alive. **Belongs in: lease/owner test plan.**

**H2. No test for the negotiation‚Üíspawn deadlock (A1).** If a viewer attach routes through the long-lived `sessionWorkflow` claim loop while a multi-day turn is running, the handshake stalls for days. No module tests viewer-attach-during-active-turn latency. **Belongs in: owner/channel-B integration tests.**

**H3. The fan-out load test (the GA gate) tests the provider tunnel ceiling but not the *control-plane* cost of N simultaneous negotiations** (N viewers each spinning a `sandboxOwnerRpcWorkflow` + acquiring a holder under one `FOR UPDATE`). The lease's single-row lock serializes all N ‚Äî a thundering-herd latency risk untested. **Belongs in: crosscut load-test plan.**

**H4. No test asserts redaction does NOT apply to Channel B and DOES to Channel A** ‚Äî the security invariant the whole acknowledgment gate rests on. The pixel plane being un-redacted is stated; nothing verifies a secret `cat`'d to the terminal appears in the framebuffer but is redacted in the `sandbox.command.output.delta` event. **Belongs in: crosscut/recording security tests.**

---

**The three highest-leverage holes:** (1) **A1/A2 ‚Äî the viewer/RPC dispatch workflow is genuinely unowned and tri-defined**; until one module owns "how does a viewer attach reach the owner," channel-A, channel-B, and recording all build on sand. (2) **A3/A5 ‚Äî the reaper SECURITY-DEFINER function + corrected `ownerHeartbeat` are the liveness/GC/billing foundation and exist only as prose pointers**, both currently broken-as-sketched. (3) **B1/B2 ‚Äî the ownership-inversion owned-session contract and the NotFound-discriminator don't compile/run as specified**, and they are the literal foundation every flow stands on.
