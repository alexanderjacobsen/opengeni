# The end-to-end pattern, grounded in real code

## A. The event spine (Channel A — structured services ride this verbatim)

**One producer → DB-of-record → bus fan-out → SSE → client projection.** Every client-visible signal in OpenGeni is an append to `session_events` that is published to NATS and fanned out over SSE. Adding a new capability = adding event types + payloads at each layer.

**1. Write+publish atom** — `packages/events/src/index.ts:30` `appendAndPublishEvents(db, bus, workspaceId, sessionId, events)`: appends via `appendSessionEvents` (DB assigns the per-session `sequence`), then `bus.publish` (`:16`) encodes a `SessionBusMessage` and `nc.publish(sessionSubject(...))`. `EventBus` type at `:7`; `subscribe` at `:36` is one NATS subscription per call. `formatSse` at `:50` emits `id:`=sequence, `event:`=type, `data:`=JSON.

**2. SSE fan-out** — `apps/api/src/http/sse.ts:5` `sseSessionStream`. Key mechanics: subscribes to the bus BEFORE replaying history (`:33`), buffers live events while `replaying` (`:34`), replays from `after` cursor via `replaySessionEvents` (`:69`), then drains the buffer. **Gap-fill** at `:20`: if a live event's sequence jumps ahead of `lastSent`, it back-fills the missing range from `listSessionEvents`. This is the durable, sequenced, gap-free, auth-per-read contract that Channel-A capabilities inherit for free — a new event type needs zero changes here.

**3. Routes + auth** — `apps/api/src/routes/sessions.ts:46`. The pattern for every route: `requireAccessGrant(c, deps, workspaceId, <permission>)` (`apps/api/src/access/index.ts:31`), then parse a contract Zod request, then `appendAndPublishEvents` and/or signal Temporal. The SSE endpoint is `:214` (`sessions:read`, reads `after`/`Last-Event-ID`). Client→agent events POST at `:296` (`sessions:control`) parse `ClientSessionEvent` and either accept a user message or `signalInterrupt`/`signalApprovalDecision`. `requirePermission`/`hasPermission` at `access/index.ts:48` (`workspace:admin` is a super-permission).

**4. Workflow signaling** — `apps/api/src/index.ts:45` `SessionWorkflowClient`. The load-bearing primitive is `signalWithStart` (`:50`, `:69`): used by `wakeSessionWorkflow` and `signalInterrupt` so a signal sent to an idle (completed) session **starts** a fresh `sessionWorkflow` that picks up the queued/buffered work, rather than throwing `WorkflowNotFoundError`. `workflowIdReusePolicy: "ALLOW_DUPLICATE"`. The Bun server (`:178`) sets `idleTimeout: 255` (max) to hold SSE connections open.

## B. The Temporal turn loop (where leases + viewer-liveness hook in)

**Workflow** — `apps/worker/src/workflows/session.ts:60` `sessionWorkflow`. Signals defined at `:43–46` (`userMessage`, `queueChanged`, `approvalDecision`, `interrupt`). The claim loop (`:82`): `continueAsNew` backstop guard (`:111`, gated by `patched("session-continue-as-new")`, fires on `continueAsNewSuggested` or `TURNS_PER_RUN_BACKSTOP=2000`, refuses while an interrupt is pending) → `claimNextQueuedTurn` (`:126`) → if empty, `maybeContinueGoal` (`:134`) then idle-wait `condition(..., "5s")` (`:156`) → `runTurn`. `runTurn` (`:195`) races the agent activity against an interrupt condition (`:223`), and branches on the result status: `requires_action` blocks on `condition(interrupt || approvalQueue)` (`:302`); `preempted` re-loops (`:293`); worker-death failures (`isWorkerDeathFailure` at `:34`, HEARTBEAT/SCHEDULE_TO_START `TimeoutFailure`) route to `requeueTurnAfterWorkerDeath` (`:254`, gated `patched("worker-death-redispatch")`).

**Critical naming detail**: the workflow schedules the **legacy** activity name `runAgentSegment` (`:215`), aliased in `apps/worker/src/activities.ts:58` to `runAgentTurn` — non-determinism-safe for multi-day in-flight histories. Any new activity the lease work adds (e.g. `ownerHeartbeat`) must be introduced behind `patched()` at a new call-site position.

**Activity registry / services singleton** — `apps/worker/src/activities.ts:29` `createActivities`. `servicesPromise` (`:30`) lazily builds `ActivityServices` once per worker process (`settings`, `db`, `bus`, `runtime`, `objectStorage`, `documentServices`, `observability`, `wakeSessionWorkflow`). **This memoized `services()` is the natural home for the in-WORKER-PROCESS `Map<sessionId, SandboxOwner>` actor registry** — it is the one per-process singleton every activity already shares.

**Worker.create** — `apps/worker/src/index.ts:33`: single `taskQueue: settings.temporalTaskQueue`, `workflowsPath`, `activities`. For per-session routing (`sandbox-owner::<sessionId>`), the design needs a second registered Worker (or dynamic queue) keyed per session; `createOpenGeniWorker` is the seam. `createWorkerWorkflowSignaler` (`:48`) shows the pattern for a worker activity to `signalWithStart` a parent workflow — the same primitive a turn-independent `ownerHeartbeat` would use to keep a viewer-only owner alive.

**The agent-turn activity** — `apps/worker/src/activities/agent-turn.ts:179` `createRunAgentTurnActivity`. Setup → `runtime.configure` → `requireSession`/`getSessionEvent` → claim/create turn → `ensureRunAllowed` (billing/limits check, `:294` and re-checked per model response at `:514`) → `startActivityHeartbeat` (`:306`) → publish `turn.started` → build agent + `runtime.runStream` (`:473`) → stream loop with `reconcileConversationTruth` (`:226`, dual-writes `session_history_items` + `upsertSandboxSessionEnvelope`) after every model response. **`upsertSandboxSessionEnvelope` (`:251`) is the existing recovery descriptor the lease/owner design extends.** Graceful-shutdown preempt path at `:638`; the `finally` at `:855` closes `preparedTools` and the heartbeat. Streaming helpers (`apps/worker/src/activities/streaming.ts`): `createRuntimeBatcher` (`:5`, flushes at 50 events / 33ms / on structural types), `startActivityHeartbeat` (`:50`, every 10s), `nextStreamEvent` (`:63`, races `iterator.next()` against `context.cancelled` so a graceful shutdown preempts mid-stream).

**Session-state activities** — `apps/worker/src/activities/session-state.ts`: `failSession` (`:32`, idempotent via `status==='failed'` short-circuit), `interruptActiveTurn` (`:69`), `requeueTurnAfterWorkerDeath` (`:118`, bounded by `WORKER_DEATH_MAX_REDISPATCHES=3`), `claimNextQueuedTurn` (`:175`), `markSessionIdle` (`:180`). The worker-death-requeue logic is the precedent for the lease design's "re-establish-singleton-from-envelope-under-CAS" recovery primitive.

## C. The contract → SDK → React → web propagation chain

A new capability's data shape is declared **once** in contracts and mirrored in 3 places (a parity test pins them):

1. **`packages/contracts/src/index.ts`** — the Zod source of truth. New event types append to `SessionEventType` enum (`:1270`). New sandbox backends append to `SandboxBackend` (`:13`, currently `docker|modal|local|none`). New client-POSTable events extend `ClientSessionEvent` discriminated union (`:1347`). New permissions append to `Permission` (`:57`). New request bodies are new Zod objects (pattern: `UpdateSessionTurnRequest` `:705`). `Session` shape at `:1234`, `SessionEvent` at `:1306`, `SessionTurn` at `:682`. `ClientConfig` (`:1425`) is how server capabilities are advertised to clients (e.g. `fileUploads`, `mcpServers`) — a desktop/headless capability flag would be surfaced here.

2. **`packages/sdk/src/types.ts`** — hand-written mirrors (zero runtime deps; `test/contract-parity.test.ts` fails on drift, per the header comment `:1`). Must duplicate: `SandboxBackend` (`:14`), `SESSION_EVENT_TYPES` array (`:100`), `KNOWN_PERMISSIONS` (`:249`), payload shapes (`:158`, e.g. `AgentToolCallCreatedPayload`). Note `SessionEventType = KnownSessionEventType | (string & {})` (`:141`) and the same open-union for permissions/usage — **older SDKs tolerate new server event types**, so a new event type is non-breaking. `packages/sdk/src/index.ts` re-exports everything; new payload types must be added to both the `export type {…}` block (`:21`) and `types.ts`.

3. **`packages/react/src/timeline.ts`** — `buildTimeline` (`:128`) is the pure event-log→renderable-items fold. A new event type = a new `case` in the `switch` (`:163`) plus a new `TimelineItem` variant in the union (`:113`). Existing variants: `UserMessageItem`, `AgentMessageItem`, `ReasoningItem`, `ToolCallItem`, `WorkerItem` (orchestration calls, `:65`), `SandboxItem` (`:79`), `SessionStatusItem`, `GoalItem`, `NoticeItem`. `groupTimeline` (`:451`) clusters activity rows. `packages/react/src/index.ts` exports hooks (`useSession`, `useSessionEvents`, `useTurnQueue`, `useGoal`, `useSessionControl`, `useFileAttachments`, …) and components — a new capability surfaces as a new hook + new timeline item kind + new component, all re-exported here. The slash-command registry (`commands/registry`, exported `:75`) is how `/clear`,`/compact`-style operator actions are wired to `sessions:control` routes.

**The exact pattern to add a client-visible capability:**
`SessionEventType` enum (contracts) → payload Zod (contracts) → `SESSION_EVENT_TYPES` + payload type (sdk/types.ts) → re-export (sdk/index.ts) → `TimelineItem` variant + `switch` case (react/timeline.ts) → hook/component (react) → worker emits via `publish([{ type, payload }])` in the agent-turn activity → API route (if client-initiated) with `requireAccessGrant` + contract parse + `appendAndPublishEvents`/signal. The SSE spine and DB are untouched.

## D. The worker / lease integration points (the ownership-inversion seam — exact code)

The architecture's "ownership inversion" is small and already half-present in the SDK types:

**The SDK already supports externally-owned sessions.** `SandboxRunConfig` (`@openai/agents-core@0.11.6/dist/sandbox/client.d.ts:60`) accepts **`session?: SandboxSessionLike`** and **`sessionState?: TSessionState`** alongside `client`. Today the runtime ignores `session` and only passes `client` + (optionally) `sessionState`:

- `packages/runtime/src/index.ts:1033` builds `runOptions`, and `:1044`–`:1048` sets `runOptions.sandbox = { client, ...(sandboxSessionState ? { sessionState } : {}) }`. The client is **built per-run** at `:1006` (`createSandboxClient(settings, environment)`) and discarded. **The inversion = (a) extend `RunAgentStreamOptions` (`:968`) with an optional externally-owned `{ session, sessionState }`, (b) at `:1044` prefer that injected session/client over `createSandboxClient`, (c) skip the per-run create/discard.** `RunAgentStreamOptions` already threads `sandboxClient` (`:969`) from `ProductionRuntimeOverrides.sandboxClient` (`:180`, `:194`) — the same override channel extends to a session handle.

- `createSandboxClient` (`:763`) is where the provider table lives: currently wires only `docker` (`:764`), `modal` (`:770`), `local` (`:791`), returns `undefined` for `none`. Adding daytona/runloop/e2b/blaxel/cloudflare/vercel = new branches here, each constructing the provider client from `@openai/agents-extensions/.../sandbox/<provider>`.

**The provider capability matrix is enforced by the SDK clients themselves:**
- **Modal** (`.../modal/sandbox.d.ts`): `ModalSandboxClient` has `resume(state)`, `serializeSessionState`, `canPersistOwnedSessionState`, `fromId(id)` (no-lock reattach), `resolveExposedPort(port): Promise<ExposedPortEndpoint>`, `tunnels()`, `supportsPty()`, `snapshotFilesystem()` (the 24h-lifetime snapshot-rollover primitive). `ModalSandboxSessionState` carries `sandboxId`, `ownsSandbox`, `imageId`, `configuredExposedPorts` — exactly the envelope fields the lease's re-establish primitive needs. **This is why R4 (2nd control handle on re-election) is safe on Modal**: `fromId` takes no lock and the control plane is per-RPC.
- **Cloudflare** (`.../cloudflare/sandbox.js:264`): `resolveExposedPort` **throws `SandboxUnsupportedFeatureError`** ("until the worker deployment exposes a port-resolution endpoint") and `:923` hard-locks `manifest.root="/workspace"` — confirming headless-only. 
- **E2B** (`.../e2b/sandbox.js:154`): `resolveRemoteExposedPort` requires the E2B SDK `getHost(port)` and returns a parsed `ExposedPortEndpoint` — desktop-capable (the noVNC port resolves to a direct host).

**The `ExposedPortEndpoint`** (`agents-core/.../session.d.ts:9`: `{ host, port, tls?, query?, protocol?, url? }`) and `urlForExposedPort(endpoint, scheme)` (`:100`, scheme `http|ws`) are exactly the Channel-B "short-lived scoped tunnel URL, direct-to-provider" primitive. `SandboxSessionState.exposedPorts` (`:36`) persists them; `recordExposedPortEndpoint`/`getRecordedExposedPortEndpoint` (`:98`) cache them in the envelope.

**The new `sandbox_leases` row mirrors an existing in-repo pattern.** `claimNextQueuedTurn` (`packages/db/src/index.ts`) already does the exact lease shape the design calls for: `withWorkspaceRls` → transaction → `SELECT id FROM session_turns … FOR UPDATE SKIP LOCKED LIMIT 1` → update. The singleton lease swaps `SKIP LOCKED` for plain `FOR UPDATE` (block, don't skip) and adds the `cold→warming` CAS + `lease_epoch` fence. The new tables (`sandbox_leases` with `uniqueIndex(workspace, session)`, `sandbox_lease_holders` one-row-per-holder) slot beside `sandboxSessionEnvelopes` (`packages/db/src/schema.ts:360`), which is already the versioned recovery descriptor `{ envelope: jsonb }` upserted per-turn. The envelope is extracted by `sandboxStateEntryFromRunState` (`packages/runtime/src/index.ts:1665`, reads `state._sandbox.sessionsByAgent[currentAgentKey]` → `{ backendId, sessionState }`) — the same blob the lease's "re-establish-singleton-from-envelope-under-CAS" reattaches from via `client.resume(state)` / Modal `fromId`.

**Viewer-only liveness** = a turn-independent `ownerHeartbeat` activity (heartbeat-TTL lease, refcount via `sandbox_lease_holders`), registered in `apps/worker/src/activities.ts` and scheduled by `sessionWorkflow` independently of `runTurn`; it uses the existing `startActivityHeartbeat` mechanism (`streaming.ts:50`) and the `wakeSessionWorkflow` `signalWithStart` primitive to keep the owner workflow alive while ≥1 viewer is attached even with no agent turn running.

## Net summary

- **Channel A capabilities** (files/git/terminal-as-events) require NO new transport: declare event type in contracts → mirror in sdk → fold in react timeline → emit from the agent-turn activity via `publish(...)`. SSE/DB/bus are unchanged. Auth-per-read is `requireAccessGrant(..., "sessions:read")` on the SSE route.
- **Channel B (desktop pixels)** lives outside the event spine: the control plane mints a scoped tunnel URL from `session.resolveExposedPort(port)` → `ExposedPortEndpoint`/`urlForExposedPort` (split-plane: negotiation/auth/lease in OpenGeni, pixels direct-to-provider).
- **The ownership-inversion seam is `RunAgentStreamOptions` (`runtime/src/index.ts:968`) → `runOptions.sandbox` (`:1044`)**: the SDK's `SandboxRunConfig` already takes `session`+`sessionState`; today the runtime builds/discards a client per run at `:1006`. Inject an externally-owned session there.
- **The lease is a new `sandbox_leases` Postgres row** following the existing `claimNextQueuedTurn` `FOR UPDATE` transaction pattern (`packages/db/src/index.ts`), with the in-process `Map<sessionId, SandboxOwner>` living in the memoized `services()` registry (`apps/worker/src/activities.ts:30`), recovery built on the existing `sandboxSessionEnvelopes` table (`schema.ts:360`) + `sandboxStateEntryFromRunState` (`runtime/src/index.ts:1665`).
- **Provider wiring is `createSandboxClient` (`runtime/src/index.ts:763`)** + the `SandboxBackend` enum (`contracts/src/index.ts:13`); the SDK clients self-enforce the desktop/headless matrix (Cloudflare `resolveExposedPort` throws, `sandbox.js:264`; Modal/E2B resolve real endpoints; Modal `fromId`/`resume`/`snapshotFilesystem` make R4 and 24h-rollover safe).
