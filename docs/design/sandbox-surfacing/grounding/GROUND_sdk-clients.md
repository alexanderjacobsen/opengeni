# SDK Sandbox Client Interfaces — Grounded from `@openai/agents-extensions@0.11.6`

All 7 providers implement the canonical `SandboxClient<TOptions, TState>` / `SandboxSession<TState>` interfaces from `@openai/agents-core@0.11.6/dist/sandbox/{client,session}.d.ts`. Source: `/home/jorge/.bun/install/cache/@openai/agents-extensions@0.11.6@@@1/dist/sandbox/<provider>/sandbox.{d.ts,js}`.

## Canonical contract (what every provider conforms to)

`SandboxClient` (client.d.ts:49): `backendId: string`; optional `create`, `delete(state)`, `serializeSessionState(state, opts)`, `canPersistOwnedSessionState(state)`, `canReusePreservedOwnedSession(state)`, `deserializeSessionState(state)`, `resume(state, opts)`.

`SandboxSession` (session.d.ts:101): `state`; optional `start/running/stop/shutdown/delete/close`, `createEditor(runAs)→Editor`, `exec`, `execCommand`, `writeStdin`, `viewImage`, `readFile`, `listDir`, `pathExists`, `materializeEntry`, `applyManifest`, `persistWorkspace()→Uint8Array`, `hydrateWorkspace(data)`, `setArchiveLimits`, `resolveExposedPort(port)→ExposedPortEndpoint`, `supportsPty()`.

`ExposedPortEndpoint` (session.d.ts:9): `{ host, port, tls?, query?, protocol?, url?, [k]:unknown }`. `ExecCommandArgs`: `{ cmd, workdir?, shell?, login?, tty?, yieldTimeMs?, maxOutputTokens?, runAs? }`. `RemoteSandboxEditor` (shared/editor.d.ts) implements `Editor` with `createFile/updateFile/deleteFile` over a `RemoteEditorIo` (mkdir/resolvePath/pathExists/readText/writeText/deletePath) — this is the file-edit/apply-patch primitive, identical across all providers.

Two base patterns: **(A)** providers extending `RemoteSandboxSessionBase` (e2b, vercel, runloop, blaxel) get exec/file/manifest/port plumbing from `shared/sessionBase.{d.ts,js}` and only implement the 6 abstract `runRemoteCommand/mkdirRemote/readRemoteText/readRemoteFile/writeRemoteFile/deleteRemotePath` + `resolveRemoteExposedPort`. **(B)** providers implementing `SandboxSession` directly (modal, daytona, cloudflare) hand-roll every method. The base `resolveExposedPort` wrapper (sessionBase.js:125) does: assert-configured → check cache (gated by `useCachedExposedPortEndpoint`) → call `resolveRemoteExposedPort` → record. `allowOnDemandExposedPorts()` controls whether ports outside `configuredExposedPorts` are permitted.

## Real vs stubbed — the load-bearing summary

**Only one hard stub exists across all 7 providers:** `CloudflareSandboxClient.resolveExposedPort` throws `SandboxUnsupportedFeatureError` unconditionally (cloudflare/sandbox.js:264-274). This is the **single decisive fact** confirming cloudflare is headless-only — no port = no VNC/desktop tunnel possible via the SDK. Everything else cloudflare offers (exec, file, editor, PTY-hardcoded-true, manifest) is real.

**Everything else marked ✅ is genuinely implemented** (real provider-SDK calls, not throws). The remaining non-✅ cells are **runtime-conditional**, not stubs:
- **PTY** is conditional on the installed provider SDK shape for e2b (`pty?.create`), daytona (`process.createPty`), blaxel (`sandboxUrl && apiKey`). If the dependency lacks the API, `execPtyCommand` throws a descriptive `SandboxUnsupportedFeatureError` at call time. Vercel + Runloop hardcode `supportsPty()=false` (no PTY by design — matters for terminal-as-events on those tiers). Modal + Cloudflare hardcode `true`.
- **Snapshot persistence** for daytona, blaxel, cloudflare is **tar-only** (real, but no provider-native FS-snapshot in the persist path — they fall through to `persistWorkspaceTar`). Modal, e2b, runloop, vercel have **native snapshot** paths with tar fallback.

**Resume / R4 ("second control handle") confirmation:** the brief's claim holds. Modal `resume()` (sandbox.js:896) calls `modal.sandboxes.fromId(sandboxId)` — a per-RPC control-plane handle with **no lock/lease** — and only throws `UserError` when `sandbox.poll()` returns a non-null exit code (box actually dead). Multiple resumes against a live box each yield an independent working session object. The other base-class providers (e2b, daytona, runloop, blaxel) route reconnect failures through `assertResumeRecreateAllowed` (shared/session.js:76): it **rethrows `UserError` as-is, returns (allowing recreate) ONLY on `isProviderSandboxNotFoundError`, and wraps everything else as `SandboxProviderError`** — i.e. recreate-on-resume happens solely when the provider reports the box genuinely gone, never on transient/conflict errors. This is the SDK primitive your "never `create()` on resume-conflict" rule layers on top of.

**Native-snapshot t(ype) encoding:** persist/hydrate use `encodeNativeSnapshotRef`/`decodeNativeSnapshotRef` (shared/snapshots) — `hydrateWorkspace` inspects the bytes; if they decode to a `{provider, snapshotId}` ref it does native restore (e.g. runloop `replaceDevboxFromSnapshot`, modal `restoreSnapshotFilesystem`/`restoreSnapshotDirectory`), else `assertTarWorkspacePersistence` + tar hydrate. This is the snapshot-rollover mechanism Modal's 24h-lifetime constraint depends on.
