# Module: OS-choice dimension (Linux now; macOS/Windows seam)  (os)

## Specification

# MODULE SPEC — OS-Choice Dimension (Linux now; macOS/Windows seam)

OpenGeni sandbox-surfacing vision. This is the implementation-grade spec for treating **OS** (`linux | macos | windows`) as a first-class dimension that threads through `session → backend+image/template selection → capability negotiation → desktop stream`. It builds ON TOP of the settled lease architecture, the split-plane data path, and the 7-provider survey — it does not re-litigate them.

> **SUPERSEDED transport note (stateless-workers ruling — see `00-master-spine.md` §B.2 and `modules/02-owner.md`).** Written before the stateless-workers ruling, parts of the body below reference an in-worker `SandboxOwner` actor and a `Map<sessionId, SandboxOwner>`. Both are **gone**: there is **no persistent owner** — any pool worker resumes the box **by id** per turn, reads `session.os` from the DB row, calls `createSandboxClient(settings, os, env)` for that turn, injects the **non-owned** handle, and drops it at turn end (file: `apps/worker/src/sandbox-resume.ts`, was `sandbox-owner.ts`). The **OS-dimension capability content is unchanged and not weakened** — OS selection is OS-aware pure functions (`selectOsImage`/`negotiateCapabilities`/shell selection) the control plane never branches on otherwise. Read "`SandboxOwner`" / "`Map<sessionId, SandboxOwner>`" below as **"the per-turn resume-by-id handle on a stateless pool worker"**; retained for provenance.

The single load-bearing fact this module exists for: **the SDK has no OS concept.** `SandboxClientOptions = Record<string, unknown>` (`/home/jorge/.bun/install/cache/@openai/agents-core@0.11.6@@@1/dist/sandbox/client.d.ts:67`), and the only OS-adjacent type in the entire SDK is the computer-use display hint `Environment = 'mac' | 'windows' | 'ubuntu' | 'browser'` (`.../dist/computer.d.ts:1`). No OS field exists on `SandboxSessionState`, `Manifest`, or any provider client options (verified: grep of `session.d.ts`/`manifest.d.ts` for `os|platform|ubuntu|windows|darwin` returns nothing). **OS is therefore an OpenGeni-owned dimension** that maps DOWN to `(backend, image/template)` at create-time and to `Environment` at computer-tool-bind-time. The SDK clients are OS-agnostic byte-pipes; OpenGeni owns the entire OS abstraction.

## 0. The provider × OS support table (the gate confirmation)

### Confirmation: all 7 first-class SDK providers are Linux-ONLY

Every one of `modal, daytona, runloop, e2b, blaxel, cloudflare, vercel` runs Linux containers (gVisor or OCI), never a macOS/Windows VM. Confirmed structurally, not by docs alone:

- **e2b**: template/image is an Ubuntu OCI image (`e2b-dev/desktop` = Ubuntu 22.04 + XFCE). `sandboxType: 'e2b' | 'e2b_code_interpreter'`, `template: string` — no OS axis.
- **modal**: runs arbitrary registry OCI image under gVisor (`ModalImageSelector.fromTag/fromImage/fromId`, `modal/sandbox.d.ts:88-90`). gVisor cannot host a foreign-kernel VM; macOS/Windows impossible by construction.
- **daytona**: `image: string` + `sandboxSnapshotName` — Linux container snapshots.
- **runloop**: `blueprintName/blueprintId` devboxes — Linux (`runloop/sandbox.d.ts:97-98`).
- **blaxel**: `image: string` (`blaxel/xfce-vnc`) — Linux.
- **cloudflare**: image baked into the Worker, `manifest.root` hard-locked to `/workspace` (`cloudflare/sandbox.js:923`) — Linux, headless-only.
- **vercel**: `runtime` is `node` | `python3.13` only, no custom image — Linux, headless-only.

**None expose any OS/platform/kernel parameter.** The Linux-only fact is structural (Linux-namespace/gVisor containers cannot run a Darwin or NT kernel), so it will not change with provider config — it requires a *categorically different* provider (a real VM host on Apple/Windows hardware).

**Decisive licensing/cost/latency reality (web-researched):**

- **macOS is legally gated to Apple hardware.** Apple's macOS SLA permits at most **2 virtualized macOS instances per physical Mac**, only on **Apple-branded hardware**, and explicitly forbids "service bureau, time-sharing, terminal sharing or other similar types of services" ([Apple SLA](https://www.apple.com/legal/sla/), [eclecticlight](https://eclecticlight.co/2022/08/04/virtualisation-on-apple-silicon-macs-8-how-apple-limits-vms/)). This kills `dockur/macos` for OpenGeni (it requires non-Apple x86 KVM, violating the EULA — its own README admits "does not permit installation on non-official hardware"). **Only AWS EC2 Mac and MacStadium/Orka are licit** because they virtualize *on real Macs*.
- **AWS EC2 Mac cost reality**: billed per-second with a **24-hour minimum Dedicated Host allocation** (Apple-SLA mandated) ([AWS EC2 Mac](https://aws.amazon.com/ec2/instance-types/mac/)). A `mac2.metal` Dedicated Host is ~$0.65–$1.08/hr but you **cannot release before 24h** → effectively ~$15–26 minimum per macOS box. This is incompatible with ephemeral per-session sandboxes; macOS must be a **pooled, pre-warmed, long-lived host fleet**, never spun per session.
- **MacStadium/Orka**: Kubernetes-style orchestration of ephemeral macOS VMs on MacStadium's Apple hardware via a REST API ([MacStadium Orka](https://macstadium.com/orka)). Monthly subscription + per-VM. Same 24h-host economics underneath but Orka pools it. This is the cleanest legit macOS path.
- **Windows is licensing-cheaper, latency-worse**: `scrapybara` offers managed Windows desktops (per-minute, ~$29–99/mo tiers, [Scrapybara](https://scrapybara.com/)); `dockur/windows` runs a Windows VM in a Linux Docker container via QEMU/KVM with a built-in noVNC web viewer ([dockur/windows](https://github.com/dockur/windows)) — legally cleaner than macOS (uses Microsoft's own trial keys), self-hostable on our own Linux fleet, but boots a full Windows VM (~30–90s cold, heavier RAM). `cua/trycua` boots Linux now, Windows/macOS "coming soon" ([Cua](https://cua.ai/)).

**Phasing verdict (honest):** v1 = Linux on the 7 SDK providers. macOS/Windows are NOT a config flag on those — they require a *new provider class*. The seam this module builds is exactly what lets a `scrapybara`/`orka`/`dockur` `SandboxClient` plug in later via a custom impl, without touching the lease/owner/event spine.

/** Pure lookup. Returns undefined for an undeclared (backend,os) pair. */
export function lookupProviderOsSupport(
  backend: SandboxBackend, os: SandboxOs,
): ProviderOsSupportRow | undefined {

  return PROVIDER_OS_SUPPORT.filter((r) => r.backend === backend && r.supported).map((r) => r.os);

## 2. SQL DDL — the OS column (no new table)

OS is a per-session/per-turn scalar, not a relation. It rides the EXISTING `sandbox_backend text` columns. Both `sessions` (`packages/db/src/schema.ts:117`) and `sessionTurns` (`:260`) get a sibling `sandbox_os text`.

```sql
-- migration: packages/db/migrations/NNNN_sandbox_os_dimension.sql

-- sessions.sandbox_os: the OS family chosen at create. Backfill linux (the only
-- OS reachable pre-migration). NOT NULL after backfill; default keeps inserts
-- from older API builds valid during rollout.
ALTER TABLE sessions
  ADD COLUMN sandbox_os text NOT NULL DEFAULT 'linux';

-- session_turns.sandbox_os: nullable per-turn override (NULL = inherit session).
ALTER TABLE session_turns
  ADD COLUMN sandbox_os text;

-- Backfill is the DEFAULT itself; existing rows become 'linux'. No data move.

-- Optional CHECK to fence the column to known values at the DB edge (defensive;
-- the Zod enum is the real gate, the column stays text to avoid enum-migration
-- churn — same posture as sandbox_backend, schema.ts:117 stores it free-text).
ALTER TABLE sessions
  ADD CONSTRAINT sessions_sandbox_os_chk
  CHECK (sandbox_os IN ('linux','macos','windows'));
ALTER TABLE session_turns
  ADD CONSTRAINT session_turns_sandbox_os_chk
  CHECK (sandbox_os IS NULL OR sandbox_os IN ('linux','macos','windows'));
```

Drizzle schema edits:

```ts
// packages/db/src/schema.ts
// sessions table (near :117 sandboxBackend)
  sandboxOs: text("sandbox_os").notNull().default("linux"),
// sessionTurns table (near :260)
  sandboxOs: text("sandbox_os"),   // nullable; inherit session when null
```

Read-side cast (mirrors the `as SandboxBackend` casts at `packages/db/src/index.ts:3549,3592`): `row.sandboxOs as SandboxOs`. **The `sandbox_leases` / `sandbox_lease_holders` tables (from the lease module) need NO os column** — the lease is `(workspace, session)`-keyed and the session row already carries `sandbox_os`; the owner reads it once on spawn. **The `sandbox_session_envelopes` table (`schema.ts:360`) likewise needs no schema change** — `os` is already implied by the provider handle in the envelope, but for the recovery-from-envelope path we additionally stamp `os` INTO the envelope JSON (§4) so re-establishment never needs a DB join.

**Migration safety / edge cases:**
- A multi-day in-flight session created before this migration has `sandbox_os='linux'` by backfill default — correct, since linux was the only reachable OS. No turn re-dispatch needed.
- An older API build (during rolling deploy) inserts a session row without `sandbox_os` → DB default `'linux'` applies. Safe.
- A rollback drops the column; sessions revert to implicit-linux (the only pre-migration behavior). Safe.

## 3. Image/template selection BY OS — the create-time mapping

This is the heart of the module: `(backend, os) → concrete image/template + exposed ports + workspace root`. It is a pure function consulted inside `createSandboxClient`.

### 3.1 The selector contract (TypeScript)

```ts
// packages/runtime/src/sandbox-os.ts  (NEW FILE)

import type { SandboxBackend, SandboxOs } from "@opengeni/contracts";
import { lookupProviderOsSupport, OS_DEFAULT_WORKSPACE_ROOT } from "@opengeni/contracts";

/** Everything create() needs that varies by (backend, os). */
export interface OsImageSelection {
  os: SandboxOs;
  backend: SandboxBackend;
  /** The provider-native image/template ref. Shape is backend-specific:
   *  - modal/docker/daytona/blaxel: an OCI image ref (string)
   *  - e2b: a template id (string)
   *  - runloop: a blueprint name (string)
   *  - cloudflare: ignored (image baked into Worker)
   *  - vercel: ignored (runtime-based) */
  imageRef: string | undefined;
  /** The single desktop stream port to expose (6080 noVNC / 3389->ws RDP),
   *  or [] for a headless (backend,os) pair. */
  exposedPorts: number[];
  /** Per-OS workspace root; generalizes the docker/modal HOME default. */
  workspaceRoot: string;
  /** SDK computer-use Environment hint to bind on the computerTool. */
  computerEnvironment: "mac" | "windows" | "ubuntu" | "browser";
}

const DESKTOP_PORT = 6080;        // noVNC (Linux Xvfb stack, §desktop-stack)
const RDP_WS_PORT  = 3390;        // websockified RDP for Windows (v3)

export class UnsupportedOsForBackendError extends Error {
  constructor(public backend: SandboxBackend, public os: SandboxOs) {
    super(`Sandbox backend "${backend}" cannot host OS "${os}". `
      + `Supported OSes: see PROVIDER_OS_SUPPORT.`);
    this.name = "UnsupportedOsForBackendError";
  }
}

/**
 * The ONE place (backend,os) -> image/ports/root is decided. Called by
 * createSandboxClient (§4). Throws UnsupportedOsForBackendError for any pair the
 * oracle marks unsupported — this is the hard fail that create-validation (§7)
 * is supposed to have already caught; reaching it means a config/DB skew.
 */
export function selectOsImage(
  backend: SandboxBackend,
  os: SandboxOs,
  settings: {
    // per-(backend,os) image overrides from config (§8). All optional;
    // undefined falls back to the curated defaults below.
    sandboxImageByOs?: Partial<Record<`${SandboxBackend}:${SandboxOs}`, string>>;
    dockerImage: string; modalImageRef?: string;
    e2bTemplate?: string; runloopBlueprintName?: string;
    daytonaImage?: string; blaxelImage?: string;
  },
): OsImageSelection {
  const row = lookupProviderOsSupport(backend, os);
  if (!row || !row.supported) throw new UnsupportedOsForBackendError(backend, os);

  const override = settings.sandboxImageByOs?.[`${backend}:${os}`];
  const computerEnvironment = os === "macos" ? "mac" : os === "windows" ? "windows" : "ubuntu";
  const workspaceRoot = OS_DEFAULT_WORKSPACE_ROOT[os];
  const exposedPorts =
    row.desktopTransport === null ? []
    : row.desktopTransport === "rdp-ws" ? [RDP_WS_PORT]
    : [DESKTOP_PORT];

  // v1: every reachable pair is linux. The curated default-image table:
  const imageRef = override ?? defaultImageRef(backend, os, settings);
  return { os, backend, imageRef, exposedPorts, workspaceRoot, computerEnvironment };
}

function defaultImageRef(backend: SandboxBackend, os: SandboxOs, s: {
  dockerImage: string; modalImageRef?: string; e2bTemplate?: string;
  runloopBlueprintName?: string; daytonaImage?: string; blaxelImage?: string;
}): string | undefined {
  if (os !== "linux") {
    // v3 seam: mac/win images are provider-class-specific (Orka template id,
    // EC2 AMI id, dockur tag). Resolved inside the custom client, not here.
    return undefined;
  }
  switch (backend) {
    case "docker":   return s.dockerImage;                 // opengeni-sandbox-desktop:linux
    case "modal":    return s.modalImageRef;               // registry desktop image tag
    case "e2b":      return s.e2bTemplate ?? "e2b-desktop"; // official XFCE+VNC template
    case "runloop":  return s.runloopBlueprintName;        // desktop blueprint
    case "daytona":  return s.daytonaImage;                // desktop snapshot/image
    case "blaxel":   return s.blaxelImage ?? "blaxel/xfce-vnc";
    case "cloudflare":
    case "vercel":
    case "local":
    case "none":     return undefined;                     // headless / image-implicit
    default:         return undefined;
  }
}
```

### 3.2 Dockerfile-by-OS (the headline-desktop image, Linux v1)

The Linux desktop image is the one we control (it is `docker/sandbox.Dockerfile` extended per the desktop-stack ground). The OS dimension means we maintain **one Dockerfile per (linux) image variant today**, with the build matrix keyed by OS for the seam. The OS dimension's contract with the build system: a per-OS image label `opengeni-sandbox-desktop:${os}` (only `:linux` exists v1). The Dockerfile path itself is selected by OS in CI: `docker/sandbox.${os}.desktop.Dockerfile`. **macOS/Windows have NO Dockerfile** — their "image" is a provider VM template (Orka template id / EC2 AMI / dockur image tag), supplied as config, never built by us. This is the honest seam: `defaultImageRef(_, "macos"|"windows")` returns `undefined`, and the custom client (§6) resolves the VM template internally.

## 4. Threading OS through `createSandboxClient` (the wiring change)

`createSandboxClient` (`packages/runtime/src/index.ts:763`) gains an `os` parameter and consults `selectOsImage`. The signature and modal/docker branches change as follows (additive — does not break the existing `none`/`local` returns):

// packages/runtime/src/index.ts  (replacing :763 signature + branches)

import { selectOsImage, UnsupportedOsForBackendError } from "./sandbox-os";
import type { SandboxOs } from "@opengeni/contracts";

export function createSandboxClient(
  settings: Settings,
  os: SandboxOs = settings.sandboxOs,               // NEW param, defaults to config
  environment = collectSandboxEnvironment(settings),

  // The (backend,os) gate. Throws for an unsupported pair (create-validation

### 5.2 The negotiation function (pure, oracle-driven)

```ts
// apps/api/src/domain/capabilities.ts  (NEW)  OR colocated in worker owner

import {
  lookupProviderOsSupport, OS_TO_COMPUTER_ENVIRONMENT, OS_DEFAULT_SHELL,
  OS_DEFAULT_WORKSPACE_ROOT, OS_PATH_SEP,
  type SandboxOs, type SandboxBackend, type SessionCapabilities,
} from "@opengeni/contracts";

/**
 * Pure capability negotiation from (backend, os) + reachability. NEVER silently
 * degrades — an unreachable desktop is transport:null, surfaced to the client,
 * which falls back to Channel-A-only. Degradation is always a handshake value.
 */
export function negotiateCapabilities(args: {
  backend: SandboxBackend;
  os: SandboxOs;
  reachable: boolean;            // is the box live / data-plane reachable?
  desktopUrl?: { url: string; token: string; expiresAt: string };
}): SessionCapabilities {
  const { backend, os, reachable, desktopUrl } = args;
  const row = lookupProviderOsSupport(backend, os);
  const supported = !!row?.supported;

  const desktopTransport = supported && reachable ? (row?.desktopTransport ?? null) : null;
  const desktopClient =
    desktopTransport === "rdp-ws" ? "web-rdp" :
    desktopTransport === "vnc-ws" || desktopTransport === "webrtc" ? "novnc" : null;

  const terminalTransport =
    !supported || !reachable ? null :
    row?.pty ? "pty-ws" : "sse-events";   // pty where the oracle allows, else events

  return {
    os,
    computerEnvironment: OS_TO_COMPUTER_ENVIRONMENT[os],
    fileSystem: {
      available: supported && reachable,
      readOnly: false,
      pathSep: OS_PATH_SEP[os],
      workspaceRoot: OS_DEFAULT_WORKSPACE_ROOT[os],
    },
    terminal: { transport: terminalTransport, shell: OS_DEFAULT_SHELL[os] },
    git: { available: supported && reachable },
    desktopStream: desktopTransport
      ? { transport: desktopTransport, client: desktopClient,
          url: desktopUrl?.url, token: desktopUrl?.token, expiresAt: desktopUrl?.expiresAt }
      : { transport: null, client: null },
    recording: { available: desktopTransport === "vnc-ws" /* x11grab only on Linux pixel plane */ },
  };
}
```

### 5.3 Headless across OS shells — the terminal-as-events path per OS

The Channel-A terminal (`sandbox.command.output.delta` events) is OS-agnostic at the transport layer but **the command construction is OS-specific**. The agent-turn activity's exec wrapper must select the shell by OS:

```ts
// apps/worker/src/activities/agent-turn.ts — where exec commands are built
// (the runtime session.exec path, runtime/src/index.ts:1242)

// linux/macos: POSIX shell with login flag
//   { cmd, shell: OS_DEFAULT_SHELL[os], login: true }   -> /bin/bash | /bin/zsh
// windows: PowerShell, no POSIX login semantics
//   { cmd, shell: "powershell.exe", login: false }
```

The SDK `ExecCommandArgs` (`{ cmd, workdir?, shell?, login?, tty?, ... }`, GROUND:sdk-clients) already carries `shell` and `login` — so terminal-as-events works across all three OSes by parameterizing those two fields from `OS_DEFAULT_SHELL[os]`. The byte stream is OS-opaque (CRLF vs LF is a client-side render concern, keyed off `capabilities.os`). **PTY (`pty-ws`) is gated twice**: the oracle's `pty` flag (false for runloop/vercel) AND the OS — Windows PTY is ConPTY-shaped (not the POSIX pty the SDK `supportsPty()` models), so v1 Windows is `sse-events`-only even though the box has a console; this is encoded by leaving Windows rows out of any `pty:true` until a Windows custom-client proves ConPTY plumbing.

## 6. The macOS/Windows seam — custom `SandboxClient` impl

This is what the whole module's abstraction exists to enable. A mac/win provider is a **custom `SandboxClient<TOptions, TState>` impl** that conforms to the same canonical interface (`/home/jorge/.bun/install/cache/@openai/agents-core@0.11.6@@@1/dist/sandbox/client.d.ts:49`) the 7 SDK clients implement — so the lease, owner, envelope, and event spine are 100% reused. It plugs in via ONE new `if` branch in `createSandboxClient`. The seam's exact contract:

```ts
// packages/runtime/src/sandbox/orka-mac.ts  (NEW, v3 — illustrative seam shape)

import type {
  SandboxClient, SandboxSessionLike, SandboxSessionState,
} from "@openai/agents-core/sandbox";

interface OrkaMacSessionState extends SandboxSessionState {
  os: "macos";
  vmId: string;            // Orka VM id (the live native-GUI Mac VM)
  templateId: string;     // Orka image template
  ownsVm: boolean;        // gates whether close() deletes
  configuredExposedPorts?: number[];   // 5900 (Screen Sharing/VNC) -> ws
}

/**
 * macOS-on-Orka SandboxClient. Conforms to the SDK SandboxClient surface so the
 * singleton-owner/lease/envelope/event-spine reuse it unchanged. Its desktop
 * stream is the VM's OWN native VNC (macOS Screen Sharing on :5900), bridged to
 * a ws URL — NOT our Xvfb stack (macOS draws to a real GUI). resolveExposedPort
 * returns the Orka-minted scoped preview URL for that port.
 */
export class OrkaMacSandboxClient
  implements SandboxClient<Record<string, unknown>, OrkaMacSessionState> {
  readonly backendId = "orka";    // <-- new SandboxBackend enum member (v3)

  async create(/* manifest, opts */): Promise<SandboxSessionLike<OrkaMacSessionState>> {
    // POST Orka REST: create VM from templateId on a pooled Apple host.
    // Return a session whose:
    //   exec()              -> SSH/ARD command channel into the Mac
    //   readFile/listDir    -> SFTP/agent-side fs
    //   resolveExposedPort  -> Orka preview URL for :5900 (native macOS VNC)
    //   supportsPty()       -> true (real zsh pty over SSH)
    throw new Error("v3 seam — not implemented in v1");
  }
  async resume(state: OrkaMacSessionState) { /* reattach by vmId, no-lock if pooled */ throw new Error("v3"); }
  async delete(state: OrkaMacSessionState)  { /* return VM to pool (respect 24h host min) */ throw new Error("v3"); }
  async serializeSessionState(s: OrkaMacSessionState) { return { ...s }; }
  async deserializeSessionState(s: Record<string, unknown>) { return s as OrkaMacSessionState; }
  // canPersistOwnedSessionState / canReusePreservedOwnedSession per pool policy
}
```

**Why the seam is clean (the load-bearing claim):**
- **Control plane unchanged.** The lease (`sandbox_leases`), the **stateless per-turn resume-by-id path** (any pool worker resumes the box by id and injects a non-owned handle — there is no in-worker `Map<sessionId, SandboxOwner>` under the stateless-workers ruling), the ownership-inversion injection point (`runOptions.sandbox = {client, sessionState}`), and the envelope-recovery primitive all operate on the *abstract* `SandboxClient` — they never branch on OS. The only OS-aware code is `selectOsImage` + `negotiateCapabilities` + the per-OS shell selection, all pure functions.
- **Channel A unchanged.** terminal-as-events / files / git ride the SAME event spine; the custom client's `exec`/`readFile`/`listDir` produce the same `sandbox.*` events. A Windows box's `dir` output and a Mac's `ls` output are identical byte-streams to the SSE fan-out.
- **Channel B is provider-native, not ours.** For mac/win the desktop stream is the VM's OWN VNC/RDP server (macOS Screen Sharing :5900; Windows RDP :3389 → websockified), reached via the same `session.resolveExposedPort(port) → ExposedPortEndpoint → urlForExposedPort` split-plane primitive. We do NOT run Xvfb/x11vnc on a Mac/Windows VM — the OS has a real GUI; we just expose its native remote-desktop port. The client picks `novnc` vs `web-rdp` from `capabilities.desktopStream.client`.
- **Computer-use unchanged in shape.** The `Computer` impl (xdotool on Linux) becomes AppleScript/`cliclick` on macOS or PowerShell SendKeys/AutoHotkey on Windows, but the SDK `Computer` interface and `computerTool({computer})` binding are identical; only `environment` differs (`'mac'`/`'windows'` vs `'ubuntu'`). The 9 methods (`click`/`type`/`screenshot`/...) map to OS-native input drivers inside the custom client's session.

**v3 provider verdicts (from research):** macOS = **MacStadium/Orka** (licit pooled Apple-HW VMs via REST, [Orka](https://macstadium.com/orka)) or **AWS EC2 Mac** (24h-min Dedicated Host — pool, never per-session). Windows = **Scrapybara** (managed, per-minute, [Scrapybara](https://scrapybara.com/)) or **dockur/windows** (self-host QEMU/KVM on our Linux fleet, built-in noVNC, [dockur/windows](https://github.com/dockur/windows)). `cua/trycua` is the one unified-API candidate spanning all three once its mac/win cloud ships ([Cua](https://cua.ai/)). `dockur/macos` is **DISQUALIFIED** — its non-Apple-HW KVM violates the macOS SLA.

## 7. Failure handling & every edge case

| # | Case | Detection | Handling |
|---|---|---|---|
| F1 | Create requests `(backend, os)` the oracle marks unsupported (e.g. `vercel + macos`) | `lookupProviderOsSupport` returns `undefined`/`supported:false` at **create-validation** (domain layer, before any provider call) | Reject create with `validation_failed` + message listing `supportedOsesForBackend(backend)`. Never reaches the worker. |
| F2 | Config/DB skew: a session row has an `(backend, os)` that became unsupported after a config change; reaches `selectOsImage` | `UnsupportedOsForBackendError` thrown in `createSandboxClient` | Owner spawn fails fast → lease `warming → cold` (caught-failure transition); session turn `failSession` with a clear OS-mismatch error. Never silently downgrades OS. |
| F3 | `os` requested but deployment's configured backend can't host it AND no alternate backend | create-validation | Same as F1 (`validation_failed`). The deployment advertises `availableOses` in `ClientConfig` so a well-behaved client never asks. |
| F4 | Desktop requested on a headless `(backend, os)` (e.g. `cloudflare + linux`, or any `vercel`) | `row.desktopTransport === null` in `negotiateCapabilities` | `desktopStream.transport = null` — handshake value, not error. Client renders Channel-A-only (terminal+files+git). The settled "degradation is always a handshake value" rule. |
| F5 | macOS/Windows requested in v1 (no custom client registered) | `createSandboxClient` falls through all branches → returns `undefined` for an `os ≠ linux` non-`none` backend | Treated as "no sandbox available" → create-validation should have blocked it (F1); if it slips through, the session has no sandbox and the agent runs Channel-A-degraded. Belt-and-suspenders: create-validation rejects any `os ∈ {macos,windows}` while `availableOses` excludes them. |
| F6 | Modal 24h lifetime expires under a live macOS-seam expectation | N/A for v1 (modal is linux-only) | Documented: macOS's analogous constraint is the AWS 24h-host MINIMUM (opposite direction — you can't release early). The owner's keep-alive must respect the per-OS lifetime floor/ceiling: Linux/Modal ceiling 24h (snapshot-rollover); macOS/EC2 floor 24h (pool, don't release). Encoded as a per-(backend,os) `lifetimeBounds` future field on the oracle row. |
| F7 | OS mismatch on resume (envelope says `macos`, current backend config is linux) | `envelope.os !== expected` AND `client.backendId` assertion (`runtime/src/index.ts:1654,1694`) | The `backendId` fence already blocks cross-provider resume; the new `os` stamp adds a cross-OS fence. | F8 | Per-turn `os` override differs from session `os` mid-session | `turn.os ?? session.os` resolution; the singleton owner already holds ONE box of `session.os` | **Per-turn OS override is REJECTED at validation when it differs from the live session's os** — a singleton box cannot change OS mid-session. `turn.os` is accepted only at the first turn (where it sets `session.os`) or when equal. Documented invariant: OS is session-immutable once a box is warm. |
| F9 | Windows `pty-ws` requested (ConPTY ≠ POSIX pty) | oracle has no `windows + pty:true` row | `terminal.transport` negotiates to `sse-events`; client gets event-terminal, not interactive pty. Honest. |
| F10 | macOS SLA 2-VM/host or pool-exhaustion on the custom client | Custom client `create` throws (Orka quota / EC2 host cap) | Surfaces as a normal provider create-failure → lease `warming → cold` → session-level retry/backoff. The OS dimension adds no new failure path here; it's a provider-capacity error like any sandbox-create failure. |
| F11 | Recording requested on non-Linux (x11grab is X11-only) | `recording.available = desktopTransport === "vnc-ws"` AND os-Linux gate | macOS/Windows recording would use native screen capture (a v3 custom-client concern); v1 `recording.available=false` off the Linux pixel plane. Handshake value. |

## 11. Honest phasing

- **v1 (Linux, ships now, headline):** `os="linux"` is the only reachable value. The full OS machinery is built (enum, column, oracle, `selectOsImage`, `negotiateCapabilities`, OS-aware shell/HOME/capabilities) but every `PROVIDER_OS_SUPPORT` row is `os:"linux"`. Desktop = our Xvfb+x11vnc+novnc stack on the 5 desktop-capable SDK providers; headless on cloudflare/vercel. `availableOses=["linux"]`. This is pure plumbing-with-one-value — zero speculative provider code.
- **v2 (the seam, no new provider):** the abstraction is exercised end-to-end with Linux only, but the custom-client INTERFACE is finalized and the `SandboxBackend` enum + `createSandboxClient` branch shape are proven against a stub `OrkaMacSandboxClient` (throws "v3"). `negotiateCapabilities` already returns `rdp-ws`/`web-rdp`/`mac` correctly for a hypothetical row. The OS-mismatch fences (F7/F8), per-OS shell selection, and envelope `os` stamp all land here so the seam is load-bearing-tested without a real Mac/Windows box.
- **v3 (macOS/Windows real):** register a custom-client provider — **macOS via MacStadium/Orka** (licit pooled Apple-HW, respects 24h-host floor) or AWS EC2 Mac (pooled); **Windows via Scrapybara** (managed) or **dockur/windows** (self-host KVM, built-in noVNC). Each is: +1 `SandboxBackend` enum member, +1 `createSandboxClient` branch, +N `PROVIDER_OS_SUPPORT` rows, +1 custom-client file, + per-OS `Computer` input driver. The lease/owner/envelope/event-spine and the entire OS-negotiation machinery are reused unchanged — that reuse is the whole point of building the dimension now. **`dockur/macos` is permanently disqualified** (non-Apple-HW KVM violates Apple's SLA).

**Key load-bearing findings:**
1. **The 7 SDK providers are confirmed Linux-only structurally** — none expose any OS/platform/kernel option; Linux-namespace/gVisor containers cannot host a Darwin/NT kernel. macOS/Windows require a categorically different provider class, not config.
2. **The SDK has no OS concept** — `SandboxClientOptions = Record<string, unknown>` (`client.d.ts:67`), and the only OS-adjacent type is the computer-use display hint `Environment = 'mac'|'windows'|'ubuntu'|'browser'` (`computer.d.ts:1`, note "ubuntu" not "linux"). OS is therefore a wholly OpenGeni-owned dimension that maps DOWN to `(backend→image, Environment)`.
3. **OS is session-immutable once warm** (F8) — the singleton box can't change OS mid-session; per-turn `os` override only sets the first turn or must equal the session OS.
4. **macOS is legally Apple-HW-only** (Apple SLA: ≤2 VMs/physical-Mac, no "service bureau"), and AWS EC2 Mac's **24h-host MINIMUM** is the inverse of Modal's 24h ceiling — macOS must be a pooled pre-warmed fleet, never per-session. `dockur/macos` is disqualified; Orka/EC2-Mac are the only licit macOS paths.
5. **The seam is genuinely clean** — a mac/win provider is a custom `SandboxClient` conforming to the same canonical interface the 7 SDK clients implement, plugging in via one `createSandboxClient` branch; lease/owner/envelope/event-spine/Channel-A are 100% reused, and Channel B becomes the VM's OWN native VNC/RDP (not our Xvfb), reached via the identical `resolveExposedPort` split-plane primitive.

## Adversarial Review

# Adversarial Review — OS-Choice Dimension Module Spec

I verified every cited line/type against the actual codebase and SDK dist. Findings are ordered by severity. Each has a concrete fix.

## BLOCKER — wouldn't compile / wouldn't run

### B1. Migration file is in the wrong directory and bypasses drizzle's journal — it will silently never apply
The spec writes `packages/db/migrations/NNNN_sandbox_os_dimension.sql`. **That directory does not exist.** Migrations live in `packages/db/drizzle/` (verified: `0000_initial.sql` … `0016_session_create_idempotency.sql`), and `drizzle.config.ts` sets `out: "./drizzle"`. Worse, drizzle tracks applied migrations via `packages/db/drizzle/meta/_journal.json` + per-migration snapshot. A hand-dropped `.sql` with no journal/snapshot entry **is never run by `drizzle-kit migrate`**. The whole DDL section (§2) is inert as written.
**Fix:** Add `sandboxOs` to `schema.ts` and run `drizzle-kit generate` to produce `0017_*.sql` + the journal/snapshot entries. If hand-writing, the file must be `packages/db/drizzle/0017_sandbox_os_dimension.sql` AND a matching `meta/0017_snapshot.json` + `_journal.json` append. The spec's narrative ("backfill via default, CHECK constraints") is fine, but the path and journal mechanics are wrong. (CHECK constraints themselves are precedented — they appear in `0001`–`0007`.)

### B2. `createSandboxClient` signature change breaks the existing call site
Current signature (`runtime/src/index.ts:763`): `createSandboxClient(settings, environment = collectSandboxEnvironment(settings))`. The spec inserts `os` as the **second positional parameter**, shifting `environment` to third. But the spec only updates the `:1006` call site partially and claims the signature "does not break the existing `none`/`local` returns" — it does break: any other positional caller, and more importantly the spec's own §4 default `os: SandboxOs = settings.sandboxOs` references `settings.sandboxOs`, which **does not exist on `Settings` yet** (it's added in §8). Ordering: if §4 lands before §8, this is a type error. Also `collectSandboxEnvironment` is imported from `@opengeni/config` (not local), unchanged — fine, but the param reshuffle is a latent footgun.
**Fix:** Make `os` an options-object field or keep it trailing/optional with an explicit default that doesn't depend on a not-yet-added settings field, e.g. `createSandboxClient(settings, environment = collectSandboxEnvironment(settings), os: SandboxOs = settings.sandboxOs ?? "linux")`. Land the `config` change (§8) **before** the runtime change (§4), or the build is red between commits.

### B3. The oracle `PROVIDER_OS_SUPPORT` references `SandboxBackend` members that don't exist
`SandboxBackend = z.enum(["docker","modal","local","none"])` (verified `contracts:13`). The oracle table (§1.2) has rows for `e2b`, `blaxel`, `cloudflare`, `vercel`, `daytona`, `runloop` as `backend:` values typed `SandboxBackend`. **These literals are not in the enum** — every such row is a TypeScript error (`Type '"e2b"' is not assignable to type SandboxBackend`). The spec's own GROUND:wiring says these 6 providers are a *separate* unshipped workstream. The spec hand-waves this by listing them in the oracle as if they're live v1 rows (and §0's matrix presents them as "SDK first-class … ✅ v1").
**Fix:** Either (a) gate the OS module behind the provider-wiring module landing first and extend the enum there, or (b) ship the oracle v1 with ONLY the 4 real backends (`docker`, `modal`, `local`, `none`) plus the not-yet-wired ones commented out exactly like the spec already does for the mac/win rows. As written, §1.2 does not type-check against HEAD.

## CORRECTNESS bugs

### C1. The docker/modal branch rewrite silently drops `dockerExposedPorts` config
Current branches use `exposedPorts: parseExposedPorts(settings.dockerExposedPorts)` (verified `:767`, `:774`). The spec replaces this with `sel.exposedPorts`, which the oracle hardcodes to `[6080]` (desktop) or `[]` (headless). This **removes the operator's ability to expose arbitrary ports** via `OPENGENI_DOCKER_EXPOSED_PORTS` and changes modal's exposed-port set (modal currently *also* reads `dockerExposedPorts` — itself a pre-existing smell, but the spec's change is a behavior change, not a no-op). The spec never reconciles `dockerExposedPorts` with the new oracle-derived ports.
**Fix:** Merge, don't replace: `exposedPorts: [...new Set([...parseExposedPorts(settings.dockerExposedPorts), ...sel.exposedPorts])]`. Document that the desktop port (6080) is now always added for desktop-capable `(backend,os)` pairs.

### C2. Per-turn `os` resolution claim is wrong about the codebase, and the override is structurally unreachable
The spec says `turn.os ?? session.os` "mirrors `turn.sandboxBackend ?? settings.sandboxBackend` (`scheduled-tasks.ts:60`)". Two problems:
1. In the **turn-execution** path, `agent-turn.ts:366` is `sandboxBackend: turn.sandboxBackend` — a **direct read, no `??` fallback**. `turn.sandboxBackend` is *required* on `SessionTurn` (`contracts:696`, non-optional). So there is no `?? session`-level fallback at execution time; the fallback the spec cites only exists in `scheduled-tasks.ts:60` for *task→turn creation*. The "mirror" is being applied at the wrong layer.
2. The spec adds `os` to `SessionTurn` as `.optional()` (§1.3) but the singleton-owner reads `session.os` **once at spawn** (§4) and the box is OS-immutable (F8). So a per-turn `os` field that differs is *rejected* (F8) and one that's equal is *redundant*. The field is therefore inert at execution time — the owner never consults `turn.os`. Adding it to `SessionTurn`/`UpdateSessionTurnRequest` is dead surface that invites the exact bug F8 must guard.
**Fix:** Drop `os` from `SessionTurn` and `UpdateSessionTurnRequest` entirely. Accept `os` only on `CreateSessionRequest` (sets `session.os` once). If a first-turn `os` channel is genuinely needed for the task→session-create path, resolve it in `createAndStartSession` into `session.os`, not onto the turn row. This also eliminates the `session_turns.sandbox_os` column (§2) — a column nothing reads.

### C3. `e2b` default image literal contradicts the grounding
§3.1 `defaultImageRef` returns `s.e2bTemplate ?? "e2b-desktop"`. GROUND:desktop-stack and §0 both say the proven template is **`e2b-dev/desktop`** (the published e2b desktop template id is `desktop`, not `e2b-desktop`). The literal `"e2b-desktop"` is invented and will fail template resolution. Same fragility for `blaxel ?? "blaxel/xfce-vnc"` — that's the *image name* from docs, plausible, but unverified as a resolvable default.
**Fix:** Use the actual e2b template id (`"desktop"` per e2b's published desktop template) or require it via config with no baked default. Don't ship an unverified template literal as a fallback. (Also moot if B3/C2 hold and e2b isn't a v1 backend.)

### C4. `OS_DEFAULT_SHELL.windows = "powershell.exe"` collides with the PTY/login model the spec itself sets up
§1.1 sets windows shell to `powershell.exe`, and §5.3 passes `{ shell, login }` into `ExecCommandArgs`. Verified `ExecCommandArgs` has `shell?` and `login?` (session.d.ts:42-43). But `login: false` for PowerShell is meaningless (PowerShell has no POSIX login shell concept), and more importantly the SDK `ExecCommandArgs.shell` is passed to a POSIX `exec` wrapper on the Linux-only providers — there is no provider in v1 that can host a Windows box, so the windows branch of `OS_DEFAULT_SHELL` is unreachable yet presented as load-bearing. Not a bug in v1 (dead code), but the spec asserts it "works across all three OSes by parameterizing those two fields" — that's untested and the `login` semantics are wrong for Windows.
**Fix:** Mark the windows/macos shell entries explicitly as v3-seam placeholders; don't claim cross-OS exec works by field-parameterization until a Windows custom client proves it. Low severity (dead in v1) but the claim is overstated.

### C5. Recording-availability gate is OS-incorrect for macOS
`negotiateCapabilities` (§5.2): `recording: { available: desktopTransport === "vnc-ws" }`. The comment says "x11grab only on Linux pixel plane," but `vnc-ws` is *also* the transport assigned to macOS in the oracle seam (`orka os:macos desktopTransport:"vnc-ws"`). So this predicate would report `recording.available=true` for a macOS box where x11grab does **not** exist (macOS Screen Sharing is not an X11 display). F11 correctly says macOS recording is false in v1, but the *code* in §5.2 contradicts F11 — it keys off transport, not OS.
**Fix:** Gate recording on `os === "linux" && desktopTransport === "vnc-ws"`, not transport alone. (The spec's prose knows this; the code doesn't implement it.)

## ARCHITECTURE / CONSTRAINT violations

### A1. `GET /sessions/:id/stream-capabilities` route shape contradicts the real routing scheme
All session routes are `/v1/workspaces/:workspaceId/sessions/:sessionId/...` (verified `routes/sessions.ts:214`). The spec's `GET /sessions/:id/stream-capabilities` omits the `/v1/workspaces/:workspaceId` prefix and uses `:id` instead of `:sessionId`. It also says `requireAccessGrant("sessions:read")` with a one-arg signature, but the real signature is `requireAccessGrant(c, deps, workspaceId, "sessions:read")` (verified `:216`). The route as specified won't mount consistently with the rest of the API.
**Fix:** `GET /v1/workspaces/:workspaceId/sessions/:sessionId/stream-capabilities`, and `await requireAccessGrant(c, deps, workspaceId, "sessions:read")`.

### A2. Envelope `os` stamp has no read path and partly duplicates the existing fence
§4.1 says stamp `os` into the envelope JSON and on re-establish call `createSandboxClient(settings, envelope.os, env)`. But the envelope today is `{ backendId, sessionState }` extracted by `sandboxStateEntryFromRunState` (verified `:1665-1683`) and round-tripped by `deserializeSandboxSessionStateEnvelope`. The `backendId` assertion (`:1654`, `:1694`) already pins the provider. Since **OS→image is fully determined by `(backend, os)` and the backend is already fenced**, and since `client.resume(state)`/`fromId` reattaches the *existing* box (it does not re-select an image), `envelope.os` is consulted only to rebuild the *client wrapper*, which is keyed off `backendId` anyway. The cross-OS fence (F7) is real only if a single `backendId` can host multiple OSes — which is exactly the mac/win custom-client case (`scrapybara` does Linux+Windows). For the 4 v1 backends (all single-OS), the `os` stamp is redundant with `backendId`.
**Fix:** Keep the `os` stamp (it's cheap and correct for the multi-OS-backend seam), but the spec should state that for v1's single-OS backends the `backendId` fence already covers F7, and the `os` fence only becomes load-bearing when a multi-OS `backendId` (scrapybara) lands. As written it overstates the v1 necessity. Minor.

### A3. `ClientConfig.availableOses` computation is under-specified for multi-backend deployments
§5.4 computes `availableOses` "from the configured backend(s) via `supportedOsesForBackend()`." But a deployment configures exactly one `sandboxBackend` (verified `config:235`, singular `SandboxBackend.default("docker")`). There is no "backend(s)" plural. And per-session/per-turn `sandboxBackend` overrides exist (`SessionTurn.sandboxBackend`), so the *reachable* OS set at create-time depends on which backend the request selects, not just the deployment default. The spec computes `availableOses` from one backend but the create path can pick another.
**Fix:** Either compute `availableOses` as the union over all backends the deployment *permits* (there's no allowlist today — another gap), or document that `availableOses` reflects only `settings.sandboxBackend` and that backend-override creates are validated independently at create-time (F1). Pick one; the spec implies both.

## GAPS — unspecified behavior / hand-waved interfaces

### G1. `sandboxImageByOs` env parser is a TODO comment, not a spec
§8: `sandboxImageByOs: /* custom parser: OPENGENI_SANDBOX_IMAGE_BY_OS k=v,k=v */`. This is the only place per-(backend,os) image overrides come from, and it's left as a comment. No grammar for the key (`"modal:linux"` collides with `:` as both the separator and the key delimiter — `modal:linux=tag` is ambiguous to a naive `split(":")`/`split("=")`), no handling of malformed entries, no validation that the backend/os in the key are valid enum members. The Zod type `z.record(z.string(), z.string())` accepts any key, so an invalid `"foo:bar"` key passes schema but `selectOsImage` never reads it (silent drop).
**Fix:** Specify the grammar (`OPENGENI_SANDBOX_IMAGE_BY_OS="modal:linux=tag1,e2b:linux=tmpl"`, split on `,` then first `=` only, key must match `/^(\w+):(\w+)$/` against the enums), reject unknown keys at config validation, and give the parser function body. The templated key type `` `${SandboxBackend}:${SandboxOs}` `` in `selectOsImage`'s `settings` param is good but unenforced at the env boundary.

### G2. `selectOsImage` `settings` param is a structurally different shape than `Settings`
§3.1 `selectOsImage(backend, os, settings: { sandboxImageByOs?, dockerImage, modalImageRef?, e2bTemplate?, ... })` takes an inline object, but §4 calls `selectOsImage(settings.sandboxBackend, os, settings)` passing the full `Settings`. `Settings` has `dockerImage` (required, verified `config:236`) and `modalImageRef?` (optional, `:240`) but **does not have** `e2bTemplate`, `runloopBlueprintName`, `daytonaImage`, `blaxelImage` until §8 adds them. So `selectOsImage`'s typed param demands fields `Settings` lacks pre-§8. Cross-module ordering hazard again, and the structural-typing means a missing field is a compile error, not a runtime undefined.
**Fix:** Type the param as `Pick<Settings, ...>` or `Partial<...>` so it's a strict subset of `Settings`, and land §8's settings additions first. Don't define a parallel ad-hoc interface that must be kept in sync with `Settings`.

### G3. F2 → "lease `warming → cold`" and `failSession` interaction is asserted, not wired
F2 says an `UnsupportedOsForBackendError` in `createSandboxClient` makes the owner "spawn fail fast → lease `warming → cold` … session turn `failSession`." But `createSandboxClient` returning/throwing is in the runtime layer (`runtime/src/index.ts`), and the lease state machine + `failSession` live in the worker/owner layer (`session-state.ts`, the unbuilt `sandbox-owner.ts`). The spec doesn't say *who catches* `UnsupportedOsForBackendError` and translates it to the `warming→cold` CAS + `failSession`. Today `createSandboxClient` is called inside `runAgentStream` (`:1006`) — a throw there propagates up through `run()`, not through any lease code. With ownership inversion the owner calls it, but the catch/translate is unspecified.
**Fix:** Specify the catch site: the `SandboxOwner.spawn()` wraps `createSandboxClient` in try/catch, maps `UnsupportedOsForBackendError` to the `warming→cold` transition + propagates a typed failure the workflow routes to `failSession`. Name the function.

### G4. No `contract-parity.test.ts` impact analysis for the new contracts
§9 says SDK hand-mirrors are fenced by `contract-parity.test.ts`. The spec adds ~10 new exported contract symbols (`SandboxOs`, `ComputerEnvironment`, `OS_TO_COMPUTER_ENVIRONMENT`, `DesktopTransport`, `ProviderOsSupportRow`, `PROVIDER_OS_SUPPORT`, `SessionCapabilities`, the maps) but only lists 3-4 as needing mirrors. The parity test (per GROUND) fails on drift — if it enumerates exports, every new const/enum needs a mirror or an explicit allowlist. The spec doesn't enumerate which of the new exports are runtime values the SDK must mirror vs. server-only.
**Fix:** Enumerate exactly which new symbols cross the SDK boundary (client-facing: `SandboxOs`, `ComputerEnvironment`, `SessionCapabilities`, `DesktopTransport`, the `OS_*` render maps for path/shell) vs. server-only (`PROVIDER_OS_SUPPORT`, `selectOsImage`, `lookupProviderOsSupport` — these should NOT ship in the SDK; the oracle is a server secret/impl detail). Confirm against the actual parity test's enumeration strategy.

### G5. `mcp/server.ts` create path not addressed
GROUND:wiring notes `apps/api/src/mcp/server.ts:528` validates `sandboxBackend: z4.string().optional()` (looser, accepts anything) — the MCP session-create surface. If sessions can be created via MCP with a backend override, they can also need an `os`, and the F1 validation must run there too. The spec's file-by-file list (§10) omits `mcp/server.ts` entirely.
**Fix:** Add `os` to the MCP create schema and route it through the same `(backend,os)` create-validation (F1), or explicitly state MCP-created sessions are always `os="linux"`.

### G6. `none` and `local` rows in the oracle assert capabilities that contradict reality
Oracle (§1.2): `local os:linux desktopTransport:null pty:true` and `none os:linux pty:false`. But `local` is `UnixLocalSandboxClient` (runs on the worker host) — marking it `supported:true, computerUse:false, pty:true` is plausible, but the spec also has `negotiateCapabilities` return `fileSystem.available=true, git.available=true` for it. For `none` (no sandbox), the oracle marks `supported:true` — that's **semantically wrong**: `none` means no sandbox at all, so `supported:true` makes `negotiateCapabilities` report a filesystem/git as available when there is none. F4/the degradation rule handle desktop, but not the "supported but capability-less" case for `none`.
**Fix:** `none` should be `supported:false` (or excluded from the oracle), so `negotiateCapabilities` returns all-`false`/null. As written, a `none` deployment advertises a working filesystem.
